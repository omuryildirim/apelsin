# Apelsin - E2E Encrypted Messaging PWA

## Project Overview

Apelsin (Russian for "orange") is a WhatsApp-like end-to-end encrypted messaging Progressive Web App. It uses a pastel orange dark theme and is designed as a mobile-first PWA with full desktop support.

## Tech Stack

### Backend (AWS CDK)
- **Location**: `aws/`
- **Runtime**: Node.js 24, TypeScript, Lambda functions
- **Database**: DynamoDB (multiple tables)
- **Storage**: S3 (private, media served via authenticated endpoint)
- **APIs**: HTTP API Gateway (REST) + WebSocket API Gateway (real-time)
- **Push**: web-push library with VAPID keys (stored in `.env`, not committed)
- **Auth**: bcryptjs for password hashing, per-device token sessions
- **CDK entry**: `bin/apelsin.ts` (loads `dotenv/config`)
- **Stack**: `lib/apelsin-stack.ts`

### Frontend (Cloudflare Workers + React)
- **Location**: `cloudflare/apelsin-fe/`
- **Framework**: React 19 + React Router 7 + Vite 8
- **Hosting**: Cloudflare Workers (SSR via `@cloudflare/vite-plugin`)
- **Styling**: Tailwind CSS v4
- **Color scheme**: Dark theme with pastel orange accents (`#1a1a2e` bg, `#22223a` cards, `#f4a261` accent, `#e76f51` hover, `#6d4c3d` own bubbles)

### Cloudflare Workers
- **`cloudflare/apelsin-api/`**: HTTP proxy to AWS API Gateway, adds CORS headers
- **`cloudflare/apelsin-media/`**: Media proxy with Cloudflare Cache API (1-year TTL), authenticates via backend verify endpoint

## 📏 Coding Standards (Strict)
- **TypeScript:** `strict` mode is on. Using `any` is an automatic rejection.
- **Functions:** Prefer `arrow functions` and `functional programming` principles wherever possible.
- **Components:** Prefer keeping the tsx files small as possible by creating smaller component that isolates functionality.
- **No duplication:** Extract shared logic into generic, reusable functions in `shared/` modules. If the same pattern appears in two places, it belongs in a shared file.
- **Single responsibility:** Each shared module has one clear purpose (e.g. `queries.ts` for DB access, `push.ts` for notifications, `call.ts` for call logic). Don't let utility files become dumping grounds.

| ApelsinMessages | chatId | sk | - | Chat messages (sk = zero-padded timestamp#uuid, optional ttl 7 days) |
| ApelsinContacts | email | contactEmail | contactEmail-index | Connection requests & accepted contacts (status: pending/accepted/declined) |
| ApelsinPushSubscriptions | email | endpoint | - | Web push subscriptions per device |
| ApelsinPairingSessions | sessionId | - | - | QR device linking sessions (ttl 5 min) |
| ApelsinSignals | to | sk | - | WebRTC signaling (ttl 5 min) |

### Chat IDs
- Format: sorted email pair joined by `__` (e.g. `alice@x.com__bob@x.com`)
- URL format: djb2 hash of chatId in base36 (e.g. `/chat/1k4m7n2`)
- Generated by `chatIdFromPair()` in `app/lib/chatIdentity.ts`

### Shared Auth (`shared/auth.ts`)
- Every request validated with 3 checks: `Authorization: Bearer <deviceToken>` + `X-Device-Id` header + `X-User-Email` header
- All must match the stored device session record
- Auth endpoints (register/login) are exempt

### Media Architecture
- **Two API Gateways**: main HTTP API + separate Media API
- **S3 bucket**: `BLOCK_ALL` public access, presigned PUT URLs for upload
- **Media serving**: Lambda streams from S3, returns base64-encoded response
- **Cloudflare media worker**: caches responses for 1 year, verifies auth+device+email+media access before serving
- **Chat media keys**: `chat-media/{chatId}/{id}` — access verified by checking email is a chatId participant
- **Profile photos**: `profiles/{email}/photo` — accessible to all authenticated users
- **Frontend**: `useAuthMedia` hook fetches with auth headers, returns blob URL

## E2E Encryption

### Key Management
- **Algorithm**: ECDH P-256 for key agreement, AES-256-GCM for message encryption
- **Key storage**: IndexedDB only (`apelsin_keys` database). NO private keys on the server.
- **Per-chat key**: `ECDH(myPrivateKey, peerPublicKey)` → shared AES key. Cached in-memory + peer public key cached in IndexedDB.
- **Multi-device**: QR-based device linking — primary device scans QR from new device, transfers encrypted private key via ephemeral ECDH channel through the server (server sees only ciphertext)

### Encryption Flow
1. Registration: generate ECDH keypair → store private in IndexedDB, public on server
2. Opening a chat: load private key from IDB → load peer's public key from IDB (or fetch from API) → ECDH derive → cache shared key
3. Sending text: `encryptMessageSymmetric(text, sharedKey)` → base64(IV + ciphertext)
4. Receiving text: `decryptMessageSymmetric(ciphertext, sharedKey)` → plaintext
5. Sending media: optimize → `encryptBlob(blob, sharedKey)` → upload encrypted bytes as `application/octet-stream`
6. Receiving media: fetch encrypted blob → `decryptBlob(data, sharedKey)` → create blob URL for display

## Features

### Messaging
- Text, image, audio messages
- E2E encrypted text
- Reply to message (quoted block, tap to scroll to original)
- Emoji reactions (6 emojis: 👍❤️😂😮😢🙏, one per user, toggle)
- Day separators in chat view
- Unread message divider
- Message pagination (100 per page, scroll-to-top loads older)
- Last message preview in chat list with smart date formatting
- Image optimization before upload (WebP)
- Audio recording (mono, 48kbps Opus)

### Real-time
- WebSocket for message notifications, presence, reactions, call signaling
- WebSocket authenticated via Lambda authorizer (token + deviceId)
- Presence: typing, recording voice, online, last seen
- All broadcasts targeted to contacts/participants only (no scan-all)

### Voice Calls
- WebRTC P2P audio via WebSocket signaling (no polling)
- Call flow: request → accept/reject → SDP offer/answer → ICE candidates → connected
- In-call controls: mute, end call
- Ring timeout (30s)
- STUN servers: Google's public STUN

### Contacts
- Add by email, requires acceptance
- Auto-accept if mutual request
- Pending requests view with accept/decline
- Only connected users can chat

### Push Notifications
- VAPID-based web push
- Service worker decrypts message text using ECDH keys from IndexedDB
- Permission requested via UI banner (not auto-prompted)
- App badge via `navigator.setAppBadge()`

### Device Management
- Per-device tokens (UUID) stored in DynamoDB with TTL
- Device fingerprinting via `deviceId` (random UUID in localStorage)
- Triple auth check: token + deviceId + email
- Active devices list in profile with revoke capability
- Logout revokes device session and clears localStorage + IndexedDB

### Multi-device
- QR-based key transfer (no private keys on server)
- New device shows QR → primary scans → ECDH encrypted key transfer
- Uses `uqr` for QR generation, `BarcodeDetector` + `jsqr` fallback for scanning

### PWA
- Full offline support via IndexedDB message cache
- Service worker for push + caching
- iOS: Safari-only enforcement (shows warning in Chrome/Firefox)
- Mobile: no logout button (moved to profile), register only on mobile
- Viewport: `maximum-scale=1, user-scalable=no` for native feel
- Swipe-to-dismiss gesture on conversation view (edge swipe)

### i18n
- English (default) + Russian
- Auto-detects from `navigator.language`
- `t("key")` / `t("key", {param: value})` pattern
- Translations in `lib/constants/en.ts` and `lib/constants/ru.ts`

## Key Design Decisions

1. **Email as identity** — no usernames, email is the PK in users table and the chat identity
2. **No private keys on server** — QR device linking instead of password-encrypted server storage
3. **Per-device tokens** — each login creates a new device session, revocable individually
4. **Triple auth** — token + deviceId + email verified on every request
5. **IndexedDB for keys** — accessible by both main thread and service worker
6. **Compound index** `[chatId, timestamp]` — O(1) cursor-based pagination regardless of chat size
7. **Separate Media API Gateway** — isolates media serving from main API
8. **WebSocket authorizer** — no unauthenticated connections possible
9. **Targeted broadcasts** — presence/messages only sent to contacts/participants, never scan-all
10. **CSS-only animations** — `SlideIn`/`FadeIn` components, no animation library dependency
